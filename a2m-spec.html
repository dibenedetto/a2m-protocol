A2M — AgentToMemory
Protocol Specification
A shared memory protocol enabling AI agents from different frameworks to read, write, and query a common memory store through a single, transport-agnostic interface.
Version0.1 (Draft)
Date2025
LicenseApache 2.0
StatusOpen for feedback

§1Abstract

The AI agent ecosystem is fragmented: every framework — LangChain, Agno, n8n, CrewAI, AutoGen — ships its own memory abstraction. Agents from different frameworks running inside the same workflow cannot exchange state, history, or knowledge without custom glue code.

AgentToMemory (A2M) defines a lightweight, wire-level protocol that decouples agents from their memory backend. Any framework implements a thin adapter; the adapter speaks A2M; any A2M-compliant store answers. No framework changes are required beyond the adapter itself.

A2M specifies a REST baseline (universally accessible), an optional WebSocket channel (reactive agents), and a pluggable storage contract covering both relational and vector backends.

§2Status of this Document

This is a Draft specification at version 0.1. It is open for review and feedback from framework authors, infrastructure providers, and early adopters. All sections are subject to change before a stable release.

Feedback should be submitted as GitHub issues at github.com/numel-ai/a2m. Breaking changes to the wire format will increment the major version number and be reflected in the URL base path.

§3Design Goals

GoalDescription
Framework agnostic No framework-specific types in the wire format. LangChain, Agno, and n8n all see the same API.
Minimal adapter surface An adapter implements four operations: write, read, query, delete. Nothing else is required.
REST baseline JSON over HTTP is the canonical transport. Any language, any runtime can participate without a dedicated SDK.
Pluggable storage The protocol does not mandate a specific database. Relational (SQLite, PostgreSQL) and vector (pgvector, Chroma) backends are both first-class.
Caller-owned embeddings A2M does not generate embeddings. The caller provides vectors; the store indexes them. This keeps the protocol model-agnostic.
Explicit namespacing Every entry is addressed by a caller-supplied namespace. No implicit scoping by session or runtime.
Idempotent writes All writes are upserts keyed on (namespace, key). Retries are safe.

§4Terminology

The key words MUST, MUST NOT, SHOULD, MAY in this document are to be interpreted as in RFC 2119.

TermDefinition
Memory Store The server-side service that persists and retrieves A2M entries. May be embedded or external.
Entry The atomic unit of memory. A key/value pair enriched with a memory type, optional embedding, and metadata.
Namespace A hierarchical address string scoping one or more entries: {app}/{workflow}/{session}/{agent}.
Embedding A caller-provided float32 vector representing the semantic content of an entry. Required for semantic query; optional otherwise.
Adapter A framework-specific wrapper that implements the framework's native memory interface and delegates all storage to the A2M REST API.
Relational Backend A SQL-compatible storage engine responsible for key lookup, metadata filtering, and TTL management.
Vector Backend A vector-index-capable engine responsible for approximate nearest-neighbour search over stored embeddings.

§5Design Decisions

The following decisions were made explicitly and constrain the rest of the specification. They are recorded here to document intent and prevent future ambiguity.

5.1Embedding Ownership

Decision: embeddings are owned and supplied by the caller. A2M stores and indexes them verbatim. The store MUST NOT generate, replace, or re-normalize embeddings.

Embedding models vary by domain, language, and task. Mandating a server-side model would couple the protocol to a specific vendor and degrade quality for callers using specialized models. By delegating embedding to the caller, A2M remains model-agnostic.

Implications:

  • Entries of type semantic SHOULD include an embedding. Semantic query on entries without embeddings is undefined behaviour.
  • The /query endpoint requires a caller-provided embedding vector. It does not accept raw text for server-side embedding.
  • Embedding dimension is not validated by the store. Callers MUST use a consistent dimension within a namespace.

5.2Namespace Resolution

Decision: namespaces are set explicitly by the caller on every request. The store performs no automatic population of namespace segments.

Automatic scoping (e.g., injecting session ID from a JWT) would require A2M to understand authentication contexts, which is out of scope. Explicit namespaces are simpler, auditable, and work across all transport layers.

Implications:

  • Framework adapters are responsible for constructing the namespace string from whatever context they have (workflow ID, session ID, agent ID, etc.).
  • The store performs only syntactic validation of the namespace (allowed characters, segment count). Semantic correctness is the caller's responsibility.

5.3Backend Pluggability

Decision: a conformant A2M store MUST provide both a relational backend (for exact lookup, metadata filtering, TTL) and a vector backend (for semantic search). These MAY be the same engine (e.g., pgvector) or separate engines.

Neither a pure SQL database nor a pure vector store is sufficient alone. Relational storage handles structured queries, key lookups, and TTL expiry. Vector storage handles semantic search. Both are required.

Reference backend pairs defined in §8:

  • Embedded / local: SQLite (relational) + FAISS (vector)
  • Managed / cloud: PostgreSQL + pgvector (single engine, both roles)
  • Decoupled: PostgreSQL (relational) + Chroma or Weaviate (vector)

5.4Write Conflict Policy

Decision: all writes are upserts keyed on (namespace, key). If an entry with the same composite key exists, it is replaced. The original id and created_at are preserved; updated_at is refreshed.

Upsert semantics make writes idempotent. Network retries, agent restarts, and at-least-once delivery do not produce duplicate entries. Callers that require an immutable append log SHOULD use unique keys (e.g., events/{timestamp_ms}).

§6Data Model

6.1Entry

An Entry is the atomic unit of memory. All fields are present in read responses. The write request body (EntryWrite) omits server-generated fields.

JSON Schema — Entry (read response) 
{
  "id":        string (UUID v7),          // server-generated, preserved on upsert
  "key":       string,                   REQUIRED  // caller-defined, unique within namespace
  "namespace": string,                   REQUIRED
  "type":      MemoryType,               REQUIRED
  "value":     any JSON,                 REQUIRED  // string, object, array, number — unrestricted
  "embedding": number[] | null,          OPTIONAL  // float32; caller-provided; required for semantic query
  "meta": {
    "source_agent":     string | null,  OPTIONAL
    "source_framework": string | null,  OPTIONAL  // e.g. "langchain", "agno", "n8n"
    "created_at":       string (ISO8601), // server-generated
    "updated_at":       string (ISO8601), // server-generated
    "ttl_s":            integer | null,  OPTIONAL  // seconds until expiry; null = permanent
    "tags":             string[],        OPTIONAL  // default []
    "confidence":       float 0–1 | null OPTIONAL
  }
}

The write request body (EntryWrite) omits id, created_at, and updated_at, which are managed by the store. All other fields follow the same schema.

6.2Memory Types

The type field classifies an entry by its role and intended lifetime. The store MAY use the type to apply default retention policies or indexing strategies.

TypeLifetimeIndexed byDescription
working Session Key In-flight scratchpad. Ephemeral task state. Typically expires at session end via ttl_s.
episodic Long Key, tags, time Interaction history. Ordered log of events. Use timestamp-based keys for natural ordering.
semantic Long Key, tags, embedding Facts and knowledge. SHOULD include an embedding; eligible for /query search.
procedural Long Key, tags Learned steps and heuristics. How to accomplish a class of tasks.
external Long Key, tags Reference pointer to an external resource — file path, URL, database row ID, blob handle.

6.3Namespace

A namespace is a slash-delimited string identifying the scope of one or more entries. The canonical four-segment form is:

Namespace format 
{app_id} / {workflow_id} / {session_id} / {agent_id}

Examples:
  numel/wf-42/sess-abc/agent-0       // single agent in a session
  numel/wf-42/sess-abc               // all agents in a session
  numel/wf-42                        // all sessions in a workflow
  numel                              // all workflows in the app

Namespace rules:

  • Each segment MUST match [a-z0-9_-]+ (lowercase alphanumeric, hyphens, underscores).
  • Trailing segments MAY be omitted to broaden scope. Reads scoped to a shorter namespace return entries from all child namespaces when recursive=true.
  • The minimum valid namespace is a single segment (app ID).
  • Maximum total length: 256 characters.

§7REST API

Base path: /a2m/v1. All request and response bodies are application/json. All timestamps are ISO 8601 UTC.

Note: the namespace is encoded as path segments in all URLs. A namespace numel/wf-42/sess-abc/agent-0 becomes the URL fragment /numel/wf-42/sess-abc/agent-0.

7.1Write (Upsert)

POST /a2m/v1/{namespace}/entries

Creates or replaces an entry identified by (namespace, key). If an entry with the same composite key exists, it is updated in place; id and created_at are preserved.

FieldTypeRequiredNotes
keystringrequiredUnique within namespace. Max 512 chars.
typeMemoryTyperequiredOne of the five defined types.
valueany JSONrequiredUnrestricted. String, object, array, or primitive.
embeddingnumber[]optionalFloat32 vector. Required for entries that will be queried semantically.
metaobjectoptionalAny subset of metadata fields. Omitted fields default to null / [].

Response: 201 Created on insert, 200 OK on update. Body: full Entry object.

Example — Write a semantic entry 
POST /a2m/v1/numel/wf-42/sess-abc/agent-0/entries

{
  "key":       "user/goal",
  "type":      "semantic",
  "value":     "Build a real-time translation pipeline",
  "embedding": [0.12, -0.04, 0.87],
  "meta": { "tags": ["user", "goal"], "confidence": 0.95 }
}

7.2Read

GET /a2m/v1/{namespace}/entries/{key}

Returns the single entry identified by (namespace, key). Returns 404 if not found.

Response: 200 OK. Body: Entry object.

7.3List

GET /a2m/v1/{namespace}/entries

Returns entries in the given namespace. Supports filtering and pagination.

Query paramTypeDefaultDescription
typeMemoryType—Filter by memory type.
tagstring (repeatable)—Filter entries whose tag list includes all specified tags.
limitinteger50Max entries per page. Max 500.
offsetinteger0Pagination offset.
recursivebooleanfalseIf true, include entries from all child namespaces.

Response: 200 OK.

JSON — List response 
{
  "entries":     Entry[],
  "total":       integer,   // total matching entries (ignoring pagination)
  "next_offset": integer | null
}

7.4Semantic Query

POST /a2m/v1/{namespace}/query

Returns the top-k most similar entries to a caller-provided embedding vector, ranked by cosine similarity. Only entries that have a stored embedding participate in the search.

MUST: the caller supplies the query embedding. The store does not accept raw text and does not perform embedding generation (see §5.1).
FieldTypeRequiredNotes
embeddingnumber[]requiredQuery vector. Must match dimension of stored embeddings in namespace.
typeMemoryTypeoptionalRestrict search to this memory type. Typically semantic.
top_kintegeroptionalNumber of results to return. Default 5, max 100.
min_scorefloat 0–1optionalMinimum cosine similarity threshold. Results below are excluded.
tagstring[]optionalPre-filter candidates by tag before vector ranking.
recursivebooleanoptionalInclude child namespaces in search space. Default false.

Response: 200 OK.

JSON — Query response 
[
  { "entry": { /* Entry */ }, "score": 0.94 },
  { "entry": { /* Entry */ }, "score": 0.81 }
]

7.5Delete

DELETE /a2m/v1/{namespace}/entries/{key}

Deletes a single entry. Returns 204 on success, 404 if not found.

DELETE /a2m/v1/{namespace}/entries?type=&tag=

Bulk delete. Deletes all entries in the namespace matching the given filters. Without filters, clears the entire namespace. Returns 204 with a X-Deleted-Count response header.

7.6Subscribe (WebSocket)

WS /a2m/v1/{namespace}/subscribe?pattern={glob}

Opens a WebSocket channel. The server sends a message whenever an entry matching pattern is written or deleted within the namespace. Pattern uses glob syntax (* matches any single segment).

JSON — WebSocket message 
{
  "event": "write" | "delete",
  "entry": { /* Entry */ }
}

§8Backend Requirements

A conformant A2M Memory Store MUST provide both a relational and a vector backend. These MAY be the same engine (e.g., PostgreSQL + pgvector) or separate engines coordinated by the store's service layer.

8.1Relational Backend

The relational backend MUST support:

  • Exact lookup by (namespace, key).
  • Prefix-based namespace filtering (for list and recursive queries).
  • Filter by type, by tag membership, by confidence range.
  • Upsert semantics preserving id and created_at.
  • TTL expiry: entries with ttl_s ≠ null MUST be expired no later than ttl_s seconds after created_at (or updated_at on upsert — implementation choice, MUST be documented).

Conceptual relational schema:

SQL — conceptual schema 
CREATE TABLE entries (
  id           TEXT         PRIMARY KEY,            -- UUID v7
  namespace    TEXT         NOT NULL,
  key          TEXT         NOT NULL,
  type         TEXT         NOT NULL,
  value        JSONB        NOT NULL,
  embedding    BYTEA        NULL,              -- float32[], backend-dependent
  source_agent TEXT         NULL,
  source_fw    TEXT         NULL,
  created_at   TIMESTAMPTZ  NOT NULL,
  updated_at   TIMESTAMPTZ  NOT NULL,
  expires_at   TIMESTAMPTZ  NULL,              -- computed from ttl_s
  confidence   REAL         NULL,
  UNIQUE(namespace, key)
);

CREATE TABLE entry_tags (
  entry_id  TEXT  REFERENCES entries(id) ON DELETE CASCADE,
  tag       TEXT  NOT NULL
);

CREATE INDEX ON entries (namespace text_pattern_ops); -- prefix scan
CREATE INDEX ON entries (type);
CREATE INDEX ON entries (expires_at) WHERE expires_at IS NOT NULL;

Reference implementations: SQLite (embedded, v1), PostgreSQL (managed).

8.2Vector Backend

The vector backend MUST support:

  • Ingestion of float32 embedding vectors keyed by entry id.
  • Approximate nearest-neighbour search (cosine similarity) returning ordered (id, score) pairs.
  • Pre-filter by namespace prefix (to restrict search space before ANN ranking).
  • Deletion of a vector by entry id.

The store's service layer joins ANN results with the relational backend to return full Entry objects.

Reference implementations: pgvector (PostgreSQL extension, co-located with relational), FAISS (embedded, local), Chroma, Weaviate (external managed).

8.3Hybrid Backend Configurations

ConfigurationRelationalVectorBest for
Embedded (v1) SQLite FAISS (in-process) Local development, single-machine deployment, zero external services.
Single-engine managed PostgreSQL pgvector Production deployments. One database, both roles, transactional consistency.
Decoupled managed PostgreSQL Chroma / Weaviate High-scale vector workloads requiring dedicated vector infra.

§9Adapter Contract

A framework adapter MUST expose at minimum the following four operations, delegating storage to the A2M REST API. The adapter is responsible for:

  • Constructing the namespace string from available framework context.
  • Providing the embedding on writes when the entry will be queried semantically.
  • Providing the embedding on query calls (not raw text).
  • Mapping the framework's native memory types to A2M memory types.
Python — Adapter interface 
from abc import ABC, abstractmethod
from typing import Any

class A2MAdapter(ABC):

    @abstractmethod
    def write(
        self,
        key: str,
        type: str,            # MemoryType literal
        value: Any,
        embedding: list[float] | None = None,
        meta: dict = {}
    ) -> dict:                # returns Entry
        ...

    @abstractmethod
    def read(self, key: str) -> dict | None:
        ...

    @abstractmethod
    def query(
        self,
        embedding: list[float],
        type: str | None = None,
        top_k: int = 5,
        min_score: float | None = None
    ) -> list[dict]:          # returns [{ entry, score }]
        ...

    @abstractmethod
    def delete(self, key: str) -> None:
        ...

Adapters MAY expose additional methods (e.g., list, subscribe) to expose the full A2M API surface to framework-native callers. The four above are the minimum required for compliance.

§10Error Codes

All error responses carry a JSON body with code (machine-readable) and message (human-readable).

JSON — Error body 
{ "code": "ENTRY_NOT_FOUND", "message": "No entry with key 'user/goal' in namespace 'numel/wf-42'" }
HTTPCodeMeaning

404ENTRY_NOT_FOUNDNo entry matches the given namespace + key.
422INVALID_NAMESPACENamespace contains invalid characters or exceeds maximum length.
422INVALID_TYPEtype is not one of the five defined MemoryType values.
422DIMENSION_MISMATCHEmbedding dimension does not match previously stored embeddings in the namespace.

§11Versioning

A2M uses a two-component version: Major.Minor.

  • Major increments on breaking wire-format changes. The major version is embedded in the URL base path (/a2m/v1/, /a2m/v2/, …). Clients and stores MAY support multiple major versions concurrently.
  • Minor increments on backward-compatible additions (new optional fields, new endpoints). Stores MUST ignore unknown fields in request bodies to remain forward-compatible.

Clients MAY include an A2M-Version: 0.1 request header to declare the spec version they were written against. Stores SHOULD use this for diagnostic logging.

Note: this document describes version 0.x (pre-stable). The major version will be set to 1 upon first stable release, at which point backward-compatibility guarantees become binding.
